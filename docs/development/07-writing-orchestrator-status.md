# 第七阶段多阶段写作编排器状态

## 本阶段目标

把现有章节续写从单次模型调用升级为多阶段写作流程，并让每个模型阶段继续使用请求级 Token 统计和自动重试基础设施。

## 已实现

- `WritingOrchestrator` 平台包装器
- 应用入口统一挂载编排器
- 项目、章节和资料库上下文构建
- 锁定人物、世界观和大纲优先
- 最近章节摘要或正文尾部接入上下文
- 模型上下文窗口容量检测
- 超出模型窗口时按优先级省略非锁定资料
- 章节规划 Prompt `chapter-planner@1`
- 正文生成 Prompt `chapter-draft-writer@1`
- 连续性联合审查 Prompt `continuity-reviewer@1`
- 定向修订 Prompt `targeted-rewriter@1`
- 审查 JSON 解析
- 低分、未通过或中高风险问题触发定向修订
- 无需修订时记录 skipped 步骤和原因
- 每个模型步骤独立统计请求、重试、Token 和延迟
- 重试后恢复到对应规划、生成、审查或改写状态
- 编排器单元测试
- 上下文容量单元测试

## 当前写作流程

```text
用户点击 AI 续写
  -> 创建 GenerationJob
  -> context_build
  -> chapter_plan
  -> draft
  -> continuity_review
  -> targeted_rewrite（按审查结果）
  -> 返回最终新增正文
  -> 现有 AppShell 保存输出并触发章节自动保存
```

## 上下文优先级

```text
项目简介
  -> 锁定大纲 / 人物 / 世界观
  -> 当前章节大纲
  -> 普通大纲 / 人物 / 世界观
  -> 最近章节摘要或正文尾部
```

只有模型上下文窗口不足时才省略非锁定资料。该行为是模型容量适配，不是 Token 限制。

## 审查与改写规则

审查报告满足任一条件时触发定向改写：

- `passed` 为 false
- `score` 低于 80
- 存在 medium 或 high 严重度问题

审查 JSON 无法解析时不进行盲目改写，而是保留初稿，并记录跳过原因。

## Token 统计

规划、初稿、审查和改写分别使用独立 `WritingStep`。每次请求和每次重试写入独立 `generation_requests` 记录。

步骤 Token 只汇总属于本步骤的请求；任务 Token 汇总所有步骤。Provider 未返回用量时保持 unknown。

## 当前限制

- 暂未分场景生成。
- 连续性、人物和风格审查仍合并在一个请求中。
- UI 中旧 Prompt 字符串仍作为兼容参数存在，但运行时不会使用。
- 保存章节与创建 AI 版本快照尚未迁入编排器。
- 记忆候选提取尚未实现。
- Token 统计管理页面尚未实现。
- 尚未使用真实 Provider 进行人工质量验收。
