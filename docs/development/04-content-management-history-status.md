# 第四阶段内容管理、版本恢复与自动重试状态

## 已实现

- 卷重命名和排序
- 章节重命名和同卷排序
- 卷与章节软删除
- 删除后的即时撤销恢复
- SQLite `deleted_at` 迁移和活动数据索引
- Web 端删除墓碑存储
- 删除卷后章节临时归入“未分卷”
- 章节版本列表
- 版本相对当前正文的字符差异摘要
- 版本内容预览
- 一键恢复章节版本
- 版本恢复后自动创建新的 `recovery` 快照
- Provider 请求失败自动重试
- 根据模型档案 `maxRetries` 限制重试次数
- 指数退避和任务重试次数持久化
- 取消生成时阻止后续重试
- 文本差异工具及单元测试

## 删除与恢复数据流

```text
用户删除卷或章节
  -> ContentRepository.delete*
  -> SQLite deleted_at / Web tombstone
  -> UI 从活动列表移除
  -> 显示撤销提示
  -> ContentRepository.restore*
  -> 恢复原始 ID、正文和版本历史
```

删除卷不会删除卷内章节。卷处于删除状态时，相关章节会显示在“未分卷”；撤销删除后会恢复原卷关系。

## 版本恢复数据流

```text
选择历史版本
  -> 计算版本与当前正文差异
  -> 用户确认恢复
  -> 覆盖章节主记录
  -> 创建 recovery 版本快照
  -> 刷新编辑器和版本列表
```

恢复操作不会破坏历史版本。每次恢复都会追加新的版本记录，可继续回退。

## 自动重试策略

- 总尝试次数为 `maxRetries + 1`
- 每次失败更新 `generation_jobs.retry_count`
- 退避时间从 400ms 开始指数增长，最大 4 秒
- 中间失败状态记录为 `retry_scheduled`
- 最终失败保留最后一次部分输出
- 用户取消后不再发起后续尝试

## 当前限制

- 删除恢复入口目前只保留最近一次删除操作；尚未实现完整回收站。
- 章节只能在当前卷内上下移动，跨卷拖拽尚未实现。
- 卷和章节排序使用整数交换，尚未实现批量归一化和拖拽排序。
- 版本差异当前是基于公共前后缀的字符摘要，不是逐段富文本 Diff。
- 自动重试复用同一个生成任务，不会为每次尝试创建独立步骤记录。
- Provider 请求仍由 WebView 运行时发起，Rust 全权代理仍待实现。
