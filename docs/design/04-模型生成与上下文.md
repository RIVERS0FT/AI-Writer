## 十、模型 Provider 架构

不要直接在组件中调用模型。

定义统一接口：

```ts id="3hy5os"
interface ModelProvider {
  testConnection(
    config: ProviderConfig
  ): Promise<ConnectionTestResult>;

  listModels?(
    config: ProviderConfig
  ): Promise<ModelInfo[]>;

  generate(
    request: GenerateRequest,
    callbacks: GenerateCallbacks
  ): Promise<GenerateResult>;

  abort(taskId: string): Promise<void>;
}
```

首期建议实现：

```text id="zvwnhp"
OpenAICompatibleProvider
AnthropicProvider
GeminiProvider
OllamaProvider
```

其中 OpenAI Compatible 可以覆盖大量服务：

- OpenAI
- DeepSeek
- 硅基流动
- 火山方舟兼容接口
- 阿里云兼容接口
- 用户自建中转
- 本地兼容服务

模型配置结构：

```ts id="076062"
interface ProviderConfig {
  id: string;
  name: string;
  providerType: string;
  baseUrl?: string;
  apiKeyRef?: string;
  customHeaders?: Record<string, string>;
}

interface ModelProfile {
  providerConfigId: string;
  model: string;
  temperature: number;
  topP?: number;
  maxOutputTokens?: number;
  contextWindow?: number;
  timeoutMs: number;
  maxRetries: number;
}
```

数据库只保存 `apiKeyRef`，不保存 API Key 明文。

## 十一、模型请求放在哪里

有两种实现方式。

### 推荐方式：Rust 发起模型请求

```text id="ewfq2u"
React
  ↓
TypeScript 构建请求
  ↓
Tauri Command
  ↓
Rust HTTP Client
  ↓
模型服务
```

优点：

- API Key 不进入 React 运行环境
- 不受浏览器跨域行为影响
- 方便统一代理、证书、超时和重试
- 更容易实现请求取消
- 更适合桌面和移动端统一处理

TypeScript 只负责生成结构化请求，Rust 负责真正发送。

### 简化方式：TypeScript 调用 Tauri HTTP 插件

适合初期原型，但 API Key 可能短暂进入 WebView JavaScript 内存。

正式版本建议将请求签名、密钥读取和网络发送放到 Rust 层。

## 十二、小说生成工作流

不要使用单次 Prompt 直接生成整本小说。

推荐分阶段执行：

```text id="vidadw"
创意输入
   ↓
题材与风格分析
   ↓
世界观生成
   ↓
人物生成
   ↓
总纲生成
   ↓
分卷规划
   ↓
章节规划
   ↓
正文生成
   ↓
一致性检查
   ↓
自动修订
   ↓
摘要与长期记忆更新
```

单章生成流程：

```text id="apjdle"
1. 读取章节目标
2. 读取当前卷大纲
3. 查询出场人物
4. 查询相关世界观
5. 查询最近章节摘要
6. 查询相关伏笔
7. 计算 Token 预算
8. 生成场景计划
9. 生成章节正文
10. 检查一致性
11. 根据检查结果修订
12. 保存章节版本
13. 提取章节摘要
14. 更新人物状态和时间线
```

任务状态：

```ts id="tsxf8s"
type GenerationStatus =
  | "queued"
  | "building_context"
  | "planning"
  | "generating"
  | "reviewing"
  | "rewriting"
  | "saving"
  | "completed"
  | "failed"
  | "cancelled";
```

任务必须持久化到数据库，避免应用关闭后完全丢失任务状态。

## 十三、上下文管理

长篇小说不能每次把全部正文发送给模型。

建议使用四层上下文。

### 第一层：永久设定

```text id="zytqje"
小说题材
写作风格
世界观核心规则
核心人物
总纲
```

### 第二层：当前结构

```text id="fiwdk1"
当前卷大纲
当前章规划
当前章目标
当前时间地点
出场人物
```

### 第三层：近期记忆

```text id="lge97s"
最近 3～5 章摘要
上一章正文末尾
人物最近状态
近期事件
```

### 第四层：检索记忆

```text id="mua2o6"
相关人物
相关地点
相关道具
相关历史事件
相关伏笔
相关章节
```

上下文构建器：

```ts id="hbhykd"
interface ContextBuilder {
  buildChapterContext(
    projectId: string,
    chapterId: string,
    tokenBudget: number
  ): Promise<ChapterContext>;
}
```

初期检索采用：

```text id="monqav"
SQLite 普通查询
+ 标签关联
+ FTS5 全文检索
```

后期再增加 Embedding 和向量检索，不建议在 MVP 阶段直接引入独立向量数据库。
